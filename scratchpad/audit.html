04 — Architecture · The Senior-Dev View

One file does 90 jobs

The codebase already has a decent module pattern — DB, Translator, Auth, Exercises are clean IIFEs talking through a global namespace. The problem is App: a 3,605-line god-module holding routing, every exercise's render+state machine, all AI features, settings, auth UI, and ~50 inline-HTML builders. It can't be tested, split across a team, or reasoned about safely.

The four structural problems

ARC-01

A single mutable state object, mutated everywhere

One state (js/app.js:405–428) mixes UI flags, current translation, and live exercise data. 100+ scattered mutation sites, with exercise code adding dynamic fields at runtime. No single source of truth, no change events → render functions are called manually after every mutation.

ARC-02

God functions blend fetch + parse + state + DOM

analyzeGermanSentence (131 ln), auditSentence (121 ln), renderWordBank (88 ln), handleAuthChange (84 ln) each own networking, JSON parsing, state writes, and inline-HTML rendering. Nothing is unit-testable.

ARC-03

~50 inline HTML string builders

Markup is concatenated in JS throughout, with manual escHtml discipline as the only XSS guard. Duplicated card/badge/result templates (e.g. auditor markup duplicated between auditSentence and renderAuditorResults).

ARC-04

Coupling via typeof X !== 'undefined' + load order

Modules reach each other through global names guarded by typeof checks scattered ~30 times. Bootstrap depends on <script> tag order in index.html — fragile and invisible.

05 — Modularization Plan

Six phases, lowest risk first

No framework, no build step required — keep the IIFE-namespace pattern the codebase already uses and split by concern. Each phase ships independently and leaves the app working. Security fixes lead because they're cheap and high-stakes.

P0Stop the bleeding — security✓ Done
  • Rotate the Gemini key & Firebase config; delete OBFUSCATED_TRIAL_KEY and the access-code array.
  • Move all Gemini calls behind /api/gemini; authenticate the proxy with the Firebase ID token; add per-uid rate limiting + a server-side trial counter.
  • Add an xp validation guard to firestore.rules (or a Cloud Function); switch XP writes to FieldValue.increment.

No refactor needed — surgical edits to gemini.js, server.js, firestore.rules, wordbank.js.

P1Extract the core: state store, router, DOM/template helpers✓ Done
  • core/store.js — wrap the state object in a tiny pub/sub store (get / set / subscribe). Replaces 100+ raw mutations with intentional updates and auto-render on change.
  • core/router.js — lift navigateTo / setupNavigation out of App.
  • core/tpl.js — a html`` tagged-template helper that auto-escapes interpolations, killing the manual escHtml footgun and de-duplicating templates.

This phase unblocks every later one. Do it before splitting features.

P2Carve out the exercise engine✓ Done
  • The biggest single win: ~1,000 lines of flashcard/MC/FITB/arrangement/match/auditor render + state machines move to features/exercises/ (one file per mode + a shared runner.js).
  • Formalize each mode's runtime data (currently dynamic fields on state) into the store under state.exercise.
P3Split translate + AI features✓ Done
  • features/translate/ — doTranslate, displayResult, synonyms, level variations, history.
  • features/ai/ — story generation, sentence auditor, verb conjugator, mistake explainer. Centralize the trial gate here as one function instead of four repeated guards.
P4Settings, sync & auth-UI modules✓ Done
  • features/settings/ — forms, theme, import/export, the localStorage↔Firestore sync logic (today tangled in handleAuthChange).
  • features/auth-ui/ — updateUserProfile, login/guest/upgrade flows, profile dropdown.
P5Optional: add a build step + testslow risk · later

  • Now-pure functions (CEFR detection, category detection, XP math, JSON cleaning, store reducers) become unit-testable. Add Vitest on those first.

Defer until the split lands — sequencing a bundler before the carve-out just moves the mess.
